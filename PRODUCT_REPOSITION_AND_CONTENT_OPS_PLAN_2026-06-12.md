# TodayPlay 产品重定位与内容运营规划方案 - 2026-06-12

## 0. 内部重审结论

本轮暂停继续做 UI 小改和 APK 打包，先重新确认产品方向。

用户反馈指出的问题是正确的：

- 首页视觉可以变好，但路线逻辑不可信，用户还是会流失。
- “武康路 + 深圳湾”这种跨城路线会直接破坏产品信任。
- 当前版本开始偏向“全球随机景点推荐器”，已经背离最初的产品初衷。

会议结论：

> TodayPlay 不是旅游攻略 App，不是陌生人社交 App，也不是纯 AI 聊天 App。它应该是“熟人关系里的今日相处脚本 App”：帮已经认识的人，在同一座城市里，把今天的见面、约会、朋友局、家庭出门、一个人散心，变成一条可信、轻松、可执行、有互动感的路线脚本。

下一版开发和美术必须先按本方案返工，不再继续堆“全球灵感”和随机 POI。

## 2. 竞品学习，不照搬的边界

### 小红书 / 内容社区

可学习：

- 首页先展示有吸引力的内容。
- 卡片要有图片、场景、标题、保存动作。
- 用户愿意为“别人已经整理好的生活灵感”停留。

不可照搬：

- 当前 TodayPlay 没有真实 UGC，不允许假装有用户笔记、热度、评论。
- 不能抓取或复刻小红书内容。

### Instagram Explore / Reels

可学习：

- 用视觉内容承接冷启动，而不是让用户先填表。
- 通过用户的点击、保存、继续看，逐步推断兴趣。

不可照搬：

- TodayPlay 不是短视频流，核心不是无限消费，而是把用户带到一次现实行动。

### Tinder / Bumble / Hinge

可学习：

- 场景入口要轻：约会、朋友、暧昧、家庭、一个人。
- Prompt 要短，降低社交尴尬。
- Bumble BFF 一类产品说明“找朋友/熟人社交”可以独立成模式。

不可照搬：

- TodayPlay 不做陌生人匹配，不做滑卡配对。
- 关系建议不能操控、冒犯或强迫亲密。

### 大众点评 / 本地生活

可学习：

- 用户相信具体城市、具体地点、地图、价格、营业状态、评价来源。
- 本地生活的信任来自数据可信度，而不是漂亮文案。

不可照搬：

- 没有授权数据时不能展示假评分、假人气、假附近、假评论。
- 不能把未核验地点包装成“热门推荐”。

### 旅游攻略 / 行程工具

可学习：

- 地点顺序、交通距离、地图预览、备选方案很重要。

不可照搬：

- TodayPlay 不是长途旅行规划。默认场景应是当天、同城、低决策成本的小路线。

## 3. 产品定位

### 一句话定位

TodayPlay 是一个面向情侣、暧昧对象、朋友、家人和独处用户的“今日相处路线脚本”App。

它解决的问题不是“去哪玩”，而是：

> 我已经知道要和谁见面，或者我今天想自己出门，但我不知道怎么安排才自然、有趣、不累、不尴尬。

### 核心承诺

3 分钟内给用户一条：

- 同城
- 可执行
- 时间合理
- 预算清楚
- 有 2-4 个地点
- 有互动任务
- 有备选方案
- 可以保存或发给同行人的今日路线

### 产品不做什么

- 不做陌生人匹配。
- 不做跨城旅游大行程。
- 不做假的本地生活平台。
- 不做未经授权的 UGC 搬运。
- 不做“AI 随便编景点”。
- 不做高风险、强亲密、操控式关系建议。

## 4. 目标用户与核心需求

### 情侣

需求：

- 不想每次都是吃饭、看电影、逛街。
- 想要一点仪式感，但不要太用力。
- 需要纪念日、下班后、周末短路线。

内容重点：

- 低压力路线
- 拍照点
- 对话提示
- 结束仪式
- 可分享纪念卡

### 暧昧对象

需求：

- 想自然升温，但怕尴尬。
- 不想安排得太像正式约会。
- 需要安全、可退、边界感强的路线。

内容重点：

- 公开场所
- 短时长
- 可随时结束
- 轻互动问题
- 不强迫亲密

### 朋友

需求：

- 周末见面不想只吃饭聊天。
- 想有一点小任务、小挑战、小笑点。

内容重点：

- Citywalk
- 小吃 / 咖啡
- 低预算
- 拍照挑战
- 随机任务

### 家人 / 亲子

需求：

- 体力压力低。
- 老人、小孩都能参与。
- 需要室内备选、休息点和路线节奏。

内容重点：

- 短距离
- 低体力
- 室内备选
- 安全舒适
- 不追求网红点

### 一个人

需求：

- 想出去散心，但不想做复杂攻略。
- 需要温柔、低成本、可暂停的路线。

内容重点：

- 公园 / 书店 / 咖啡 / 河边
- 低预算
- 情绪恢复
- 给自己的记录任务

## 5. 内容产品形态

## 6. 路线规划硬规则

这是开发必须先实现的规则，不允许 AI 或 UI 绕过。

### P0 规则：同城约束

一条路线只能属于一个城市。

禁止：

- 上海武康路 + 深圳湾
- 广州老街 + 杭州西湖
- 北京故宫 + 成都太古里

生成器必须做到：

1. 先确定 routeCity。
2. 只从 routeCity 的 POI 池里选地点。
3. 如果该城市 POI 不足，降低站点数量或提示该城市内容暂未覆盖。
4. 不允许用其他城市 POI 补齐。

### P0 规则：城市显示必须明确

首页每张路线卡必须展示具体城市，不允许再用模糊的“全球热门城市”作为可生成路线。

可以展示：

- 上海 · 武康路 90 分钟散步
- 深圳 · 深圳湾日落短线
- 广州 · 老街小吃朋友局

不可以展示：

- 全球热门城市
- 全球灵感
- 今日推荐，但生成时没有固定城市

### P0 规则：AI 不能发明地点

无论未来接 Kimi API 还是其他模型，AI 都只能从经过校验的候选 POI 里选择和组织。

AI 不允许：

- 自己编地点
- 自己编地址
- 自己编营业时间
- 自己编评分
- 自己编热度
- 跳过同城校验

### P1 规则：路线距离和时长合理

MVP 阶段可以先使用简化规则：

- 90 分钟路线：2 个地点，同区或近邻区域优先。
- 2 小时路线：2-3 个地点，移动时间不超过 25 分钟。
- 半日路线：3-4 个地点，移动时间不超过 45 分钟。
- 一天路线：正式版再开放，当前版本不建议主推。

### P1 规则：关系场景安全

暧昧、情侣、独处场景要避免：

- 过于偏僻
- 夜间无人区域
- 强迫亲密任务
- 让用户尴尬的公开表演
- 不合适的消费压力

### P1 规则：内容来源状态

每条路线必须有内容可信度状态：

- 本地编辑样例
- 官方/授权数据
- 商家合作
- 用户自建
- 待核验

待核验内容不能显示“热门”“附近”“评分”“推荐榜”这类强背书词。

## 7. AI / Kimi API 接入策略

AI 可以用，但不能替代规则系统。

### 正确分工

规则系统负责：

- 城市锁定
- POI 候选池
- 距离/时长硬约束
- 安全过滤
- 来源状态
- 结果结构校验

Kimi API / AI 负责：

- 把候选地点组织成更自然的路线文案
- 生成互动任务
- 生成对话提示
- 生成分享卡文案
- 根据用户语气偏好调整风格
- 在已有候选路线中做解释和排序

### 技术接入原则

- API Key 不能写进 Android 客户端。
- Android 只调用自己的后端。
- 后端代理 Kimi API。
- 请求必须使用结构化 JSON。
- 返回必须再过服务端校验。
- 客户端只展示校验后的路线。

### 推荐 Kimi 请求结构

输入：

- routeCity
- relationshipType
- duration
- budget
- moodTags
- candidatePois
- forbiddenPoiIds
- safetyRules
- outputSchema

输出：

- routeTitle
- selectedPoiIds
- stopOrder
- stopReason
- interactionTask
- photoTask
- backupPlan
- shareCopy

服务端校验：

- selectedPoiIds 必须全部在 candidatePois 中。
- selectedPoiIds 必须全部同 cityId。
- stopOrder 数量必须符合 duration。
- 不通过则丢弃 AI 结果，回退本地规则生成。

## 8. 首页与交互设计方案

### 首页第一屏

首页不再以“全球灵感”为主，而是以“今天你在哪座城市，和谁出去”为主。

结构：

1. 顶部：
   - TodayPlay
   - 当前城市
   - 设置入口

2. 场景：
   - 约会
   - 暧昧
   - 朋友
   - 家庭
   - 一个人
   - 雨天
   - 低预算

3. 内容流：
   - 同城路线卡
   - 城市清晰可见
   - 图片 / 地图感视觉
   - 适合关系
   - 时长
   - 预算
   - 开始 / 收藏 / 邀请

4. 轻提示：
   - “当前为本地样例路线，正式版会接官方地图和授权内容源。”

### 路线卡片

卡片必须展示：

- 城市
- 区域
- 标题
- 适合关系
- 总时长
- 预算
- 2-3 个路线点摘要
- 是否已核验

卡片不展示：

- 假评分
- 假热度
- 假附近
- 假用户头像
- 不确定来源的图片

### 路线详情页

点击卡片后，不应直接生成完整结果，可以先出现轻详情：

- 路线地图预览
- 站点顺序
- 预计移动方式
- 为什么适合今天
- “开始这条路线”
- “换一个更轻松/更便宜/更适合拍照的版本”

### 结果页

结果页继续保留：

- 路线预览
- 当前站
- 下一步行动
- 打卡
- 替换一站
- 恢复上一站
- 分享卡

但必须新增：

- 城市一致性说明
- 内容来源状态
- “地点信息请以官方地图为准”

## 9. 内容运营规划

### 阶段 1：样例城市包

先不要做“全球热门城市”。

建议先做 4 个城市：

- 上海
- 深圳
- 广州
- 杭州

每个城市先做 8 条路线脚本包：

- 情侣下班 90 分钟
- 暧昧低压 90 分钟
- 朋友低预算 2 小时
- 一个人散心 90 分钟
- 家庭低体力半日
- 雨天室内备用
- 拍照路线
- 纪念日路线

总量：4 城市 x 8 条 = 32 条高质量样例路线。

### 阶段 2：内容生产流程

每条路线脚本包必须经过：

1. 选题
2. 城市和区域确认
3. 地点收集
4. 同城/距离校验
5. 关系场景适配
6. 安全和边界审查
7. 文案生成
8. 人工复核
9. QA 真机路径测试
10. 上线

### 阶段 3：内容来源策略

MVP 可用：

- 本地编辑样例
- 官方地图手动核验后的地点名和地址
- 自有拍摄或生成图片
- 用户自建路线

正式版需要：

- 官方地图 API
- 授权 POI 数据
- 商家合作内容
- 后端内容管理系统
- 用户授权上传内容

禁止：

- 抓取第三方平台内容
- 未授权使用商家图片
- 展示虚假热度、评分、评论

### 阶段 4：内容质量指标

内容运营每周看：

- 首页卡片点击率
- 路线开始率
- 收藏率
- 分享率
- 路线完成率
- 替换站点率
- 路线投诉率
- 城市错误率
- “看不懂/不想去”反馈率

P0 指标：

- 城市错误率必须为 0。
- 跨城路线不得进入构建包。

## 10. 下一版返工范围

### 产品经理任务

- 冻结新定位：熟人关系里的今日相处脚本。
- 删除“全球随机路线”作为默认产品承诺。
- 首页文案从“全球灵感”改为“选择城市和今天的关系场景”。
- 定义 32 条样例路线脚本包的内容表。

### 内容运营任务

- 建立城市包内容表。
- 先补上海、深圳、广州、杭州。
- 每条路线写明城市、区域、适合关系、时长、预算、站点顺序、备选方案。
- 武康路只能进入上海路线。
- 深圳湾只能进入深圳路线。

### 开发任务

P0：

- 给 POI 增加 canonical cityId 或 normalizedCity。
- 生成器先锁 routeCity。
- 候选 POI 必须按 routeCity 过滤。
- 候选不足时减少站点或提示暂未覆盖。
- 增加 `sameCityRouteValidator`。
- 回归检查必须验证不存在跨城市路线。

P1：

- 首页卡片必须绑定具体城市。
- 移除“全球热门城市”作为可生成路线输入。
- 新增路线轻详情页。
- 收藏 tab 做成本地收藏列表，而不是跳转到 Quick Start。

P2：

- 设计 Kimi API 后端代理结构。
- AI 输出必须使用候选 POI ID，不允许自由编地点。

### 美术 UI 任务

- 首页卡片视觉从“全球灵感”改成“城市路线包”。
- 卡片上城市和区域必须明显。
- 增加地图感路线缩略图。
- 设计“已核验 / 本地样例 / 待核验”视觉标签。
- 路线详情页要比结果页更轻，帮助用户先判断是否想开始。

### QA 任务

新增 P0 测试用例：

- 上海路线不能出现深圳 POI。
- 深圳路线不能出现上海 POI。
- 选择武康路相关卡片，结果页全部站点必须是上海。
- 选择深圳湾相关卡片，结果页全部站点必须是深圳。
- 首页所有可开始卡片必须有明确城市。
- 没有覆盖的城市不能生成假路线。

## 11. 下一版验收标准

V0.9.47 或下一轮实现版本必须满足：

- 首页不再出现“全球热门城市”作为可直接生成路线。
- 每张路线卡有明确城市。
- 每条生成路线所有 POI 城市一致。
- 跨城路线自动检查失败时构建不可通过。
- 路线结果页展示城市和来源状态。
- Kimi/API 如接入，只能作为文案和排序辅助，不能绕过 POI 校验。

## 12. 最终决策

开发和美术暂停继续围绕 V0.9.46 的瀑布流做细节美化。

下一步先做“内容可信度和路线同城性返工”。

产品主线改为：

> 同城今日相处脚本，而不是全球景点生成。

只有这条主线成立，首页瀑布流、图片、地图、分享、AI 生成才有意义。
