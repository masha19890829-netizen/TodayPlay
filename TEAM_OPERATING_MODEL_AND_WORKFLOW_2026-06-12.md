# TodayPlay 团队介绍与工作流规范 - 2026-06-12

## 1. 团队定位

TodayPlay 当前团队的目标不是单纯“把 APK 打出来”，而是把这个应用持续迭代成一个用户愿意打开、能快速理解、路线可信、内容可扩展、最终具备上架条件的移动应用。

经过 V0.9.27 到 V0.9.51 的连续迭代，团队已经形成一条核心经验：

> 先证明核心体验成立，再补账号、支付、上架和商业化。

当前产品方向已经收敛为：

> TodayPlay 是一个“同城关系路线脚本 App”，帮助情侣、暧昧对象、朋友、家人和独处用户，把今天的见面或出门变成一条可信、轻松、可执行、可分享的路线。

它不是陌生人社交，不是跨城旅游攻略，不是假本地生活平台，也不是纯 AI 随机景点生成器。

## 2. 已沉淀的关键经验

### 2.1 用户反馈优先级高于原方案

用户指出“文字太多、没有图片、没有地图、没有推荐、路线跨城不可信”时，团队必须立即把这些视为产品方向问题，而不是单个 UI 小问题。

经验规则：

- 用户一打开没兴趣，是 P0 体验问题。
- 路线不可信，是 P0 信任问题。
- 首页把设置项放到推荐流前面，是 P1 信息架构问题。
- 上架、支付、Google 登录是重要外部依赖，但不能掩盖核心体验未成立。

### 2.2 前 2 分钟决定留存

新用户不是来读说明书的。前 2 分钟应该让用户看到可玩的路线卡片、做一个轻选择、获得一个可执行结果。

经验规则：

- 首页先给内容，不先给表单。
- 首页先给视觉卡片，不先给长文案。
- 登录、语言、隐私、支付都不应该阻塞浏览和第一次生成。
- 用户先看到价值，再被邀请登录或个性化。

### 2.3 路线可信度是底线

武康路在上海，深圳湾在深圳，这类跨城路线会直接破坏用户信任。路线逻辑必须先锁定城市，再选 POI，再生成文本。

经验规则：

- 每条路线必须属于一个具体城市。
- 同一天路线不允许跨城市。
- 卡片上要提前展示城市、站点、站数和同城校验。
- AI 可以润色和辅助排序，但不能绕过城市、距离、时间、安全这些硬规则。

### 2.4 内容运营要结构化

早期把内容写在首页 UI 里，速度快但不可持续。V0.9.50 到 V0.9.51 已经开始把内容抽到 `HomeRouteContentCatalog`。

经验规则：

- 城市包、路线包、站点预览、内容状态应该进入数据层。
- UI 负责展示，不负责维护运营内容。
- 每条内容应标注来源状态：本地样例、运营样例、待核验、已核验、外部授权等。
- 后续接 Kimi API 或本地模型时，也应先输出结构化内容，再进入 UI。

### 2.5 竞品调研要学习机制，不照搬表面

团队从内容社区、社交发现、本地生活、约会、旅行规划类产品中学到的是机制，不是复制内容。

可学习：

- 内容流先吸引用户。
- 分类入口降低选择压力。
- 卡片要有图片、标题、标签、保存、开始、邀请。
- 本地生活要强调可信地点、地图、价格、距离和来源。
- 社交产品要把关系意图拆清楚。

不可照搬：

- 不复制竞品文案、UGC、评分、热度。
- 不伪装“附近热门”。
- 不做陌生人匹配滑卡。
- 不把 TodayPlay 做成跨城旅游大攻略。

### 2.6 每版必须有可验证交付

从 V0.9.45 之后，团队开始形成固定交付纪律：每次版本改动都要有版本号、APK、报告、构建验证、模拟器或设备验证。

经验规则：

- 没有验证的版本不能交给用户试。
- 不能只说“我改了”，必须说明怎么验过。
- 模拟器启动和日志检查是基础门槛。
- 对路线类功能，必须查结果页是否出现错误城市。

## 4. 标准工作流

### 4.1 接收反馈

项目负责人先把用户反馈翻译成产品问题。

示例：

- “打开没兴趣” -> 首屏价值表达失败。
- “文字太多” -> 信息密度和视觉层级失败。
- “武康路和深圳湾放一起” -> 路线可信度 P0 失败。
- “调研组没起作用” -> 竞品学习没有转成可执行设计。

### 4.2 内部评审

每次大方向变动前，至少让这些角色参与判断：

- 首次体验产品经理：用户前 2 分钟应该先做什么。
- 竞品与用户动机研究员：竞品有哪些机制可学，哪些不能碰。
- 关系场景内容策划：内容包应该服务哪个关系场景。
- UI 动效设计师：第一屏、卡片、结果页怎么更像现代社交发现产品。
- Android 开发工程师：最小可交付版本怎么做。
- QA：验收怎么证明没有跑偏。

### 4.3 方案输出

方向调整必须先出方案，再开发。

方案至少包含：

- 产品定位
- 目标用户
- 首屏结构
- 前 2 分钟路径
- 内容包结构
- UI 风格方向
- 路线逻辑底线
- 本轮验收标准

### 4.4 版本开发

开发只做一个清晰切片，不把所有想法塞进同一版。

版本开发要求：

- 升级 `versionCode` 和 `versionName`。
- 代码改动要围绕用户路径。
- 内容尽量进结构化目录，不写死在 UI。
- 登录、支付、外部 API 不允许硬编码密钥。
- 无真实后端验证前，支付和 Google 登录只能作为安全脚手架。

### 4.5 验证测试

每次交付 APK 前至少执行：

- 产品回归审计。
- Debug 构建。
- Lint / 测试任务。
- Release 构建。
- APK 元数据检查。
- APK 签名检查。
- SHA-256 哈希记录。
- 模拟器或真机安装。
- 冷启动。
- 首屏截图。
- 主路径点击到结果页。
- 最近日志崩溃检查。

路线版本额外检查：

- 结果页城市正确。
- 站点属于同一城市。
- 不出现跨城组合。
- 地图/路线预览可见。
- 当前站和下一步动作清晰。

### 4.6 版本报告

每个版本必须输出更新报告：

- 本轮目标
- 已完成改动
- 产品负责人判断
- 验证结果
- APK 和截图路径
- APK 哈希
- 下一轮建议

## 5. 验收分级

### P0 阻断

出现以下问题不能交付：

- 打开闪退。
- 首屏无法理解产品价值。
- 首页只剩文字墙。
- 路线跨城市或明显不可能执行。
- 支付在无验单后端时可真实购买。
- Google 登录未配置后端验证却被当作可信账号。
- 使用假评分、假热度、假用户评论。

### P1 高优先

应尽量在当前版本修复：

- 推荐卡没有图片或路线预览。
- 首页把设置项、语言选择、长表单放在前面。
- 结果页缺地图感和下一步行动。
- 内容包没有来源或状态标注。
- 分享、收藏、开始入口不明显。
- 模拟器未测主路径。

### P2 打磨

可以排入后续：

- 动效反馈更细。
- 卡片文案更短。
- 截图更适合商店展示。
- 更多城市包。
- 更多关系场景。
- 更丰富的完成奖励和回访机制。

## 8. 下一步团队建议

1. 内容策划继续扩城市包：上海、深圳、成都、重庆、南京、厦门，每城至少 3 条关系场景路线。
2. UI 动效设计师继续优化城市包卡片，让“运营精选 / 待核验样例 / 本地样例 POI”更易懂。
3. 路线逻辑工程师准备 Kimi API 接入方案，但必须保留本地硬约束和无 API 兜底。
4. QA 把三频道路径都纳入自动检查：今天可做、城市包、关系包。
5. 发布负责人继续等待真实外部资料：开发者账号、隐私政策 URL、Google OAuth、支付验单后端、商店素材。

## 9. 项目负责人结论

团队已经具备持续开发 APP 的基本经验：能听用户反馈、能重定产品方向、能拆角色、能打包、能模拟器验证、能把经验变成流程。

下一阶段要避免“为了功能而功能”。每个版本都必须回答一个问题：

> 用户打开 TodayPlay 后，是否更快、更安心、更愿意开始一条真实可执行的同城路线？
