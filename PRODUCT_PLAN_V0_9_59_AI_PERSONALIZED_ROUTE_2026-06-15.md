# TodayPlay V0.9.59 AI 个性化路线内测版计划

日期：2026-06-15  
版本目标：`0.9.59 / versionCode 77`  
版本代号：`AI Personalized Route Preview`

## 一句话目标

把 TodayPlay 从“固定样例路线推荐”升级成“用户说出今天的状态后，生成一条真正贴合当下需求的同城路线”。

V0.9.59 的核心不是堆更多卡片，而是补上产品的灵魂：

```text
你告诉我今天和谁、在哪、什么心情、预算和体力；
我给你一条有理由、能执行、同城、不尴尬、可撤退的路线。
```

## 用户问题

当前 V0.9.58 的主要问题：

1. 内容像固定剧本。
   不同用户点进去，路线和文案差异有限。

2. 用户输入没有真正影响结果。
   频道和路线卡有变化，但用户自己的状态没有被深度使用。

3. 缺少“被理解”的瞬间。
   App 看起来更漂亮了，但还不像一个会帮我思考今晚安排的人。

4. 结果页和首页卡片有时不连续。
   用户点了一条卡，结果页标题和叙事可能变成另一套说法。

5. 收藏、反馈、历史还没有形成闭环。
   外测用户喜欢或不喜欢一条路线后，没有足够明确的回传机制。

## 版本定位

V0.9.59 不做正式上线级 AI，不做无限开放生成。

本版做：

- AI 个性化生成的内测入口。
- Kimi / DeepSeek 可切换的后端模型网关设计。
- 用户一句话需求解析。
- 同城候选 POI + 规则约束 + AI 文案生成。
- 本地生成器兜底。
- 外测反馈入口。

本版不做：

- 不把大模型 API Key 放进 Android 客户端。
- 不让模型凭空编造 POI。
- 不接真实地图 SDK。
- 不接支付。
- 不做公开宣传“AI 已完全上线”。
- 不做无限城市真实数据承诺。

## 模型接入策略

### 推荐方向

采用“后端模型网关 + Android 调用自家 HTTPS endpoint”的方式。

原因：

- API Key 不能放进 APK。
- 需要控制成本、频率、超时和风控。
- 需要统一 Kimi / DeepSeek / 本地兜底。
- 需要记录失败原因和模型输出质量，方便外测复盘。

### Provider 选择

优先建议：

1. Kimi / Moonshot
   - 更适合作为中文生活方式路线文案的第一候选。
   - 官方文档显示可用 OpenAI SDK 兼容方式调用，`base_url` 为 `https://api.moonshot.cn/v1`。

2. DeepSeek
   - 适合作为成本友好、可切换的第二候选。
   - 官方文档提供 Chat Completion API、Bearer Auth、OpenAI API 格式示例。

### 不直接二选一写死

V0.9.59 应该做模型抽象：

```text
Android
  -> TodayPlay AI Route Gateway
      -> Provider: Kimi
      -> Provider: DeepSeek
      -> Fallback: LocalRouteGenerator
```

这样以后用户给 Kimi Key、DeepSeek Key 或别的模型，都可以在后端切换，不需要重写 App。

## AI 生成原则

### 模型负责什么

模型负责：

- 理解用户一句话需求。
- 把情绪转成路线偏好。
- 给候选路线排序。
- 生成更贴近用户的标题、副标题、路线理由。
- 为每个站点生成轻任务、聊天提示、撤退方案。
- 生成邀请文案和通关卡文案。

### 模型不负责什么

模型不负责：

- 凭空发明地点。
- 决定城市跨度。
- 绕过预算/时间/移动压力限制。
- 使用未授权平台数据。
- 生成危险、过度亲密、冒犯或不适合关系阶段的任务。

### 硬约束必须由程序控制

无论模型怎么说，程序必须强制：

- 只选一个城市。
- 只从候选 POI 池选择站点。
- 站点数 2-4 个。
- 总时长不超过用户选择。
- 预算不超过用户选择。
- 移动压力符合用户体力。
- 约会/朋友/家人/独处的任务边界不同。
- 夜间路线避免偏僻、高风险或不可撤退场景。

## 产品体验改造

## 1. 首页新增“说一句今天”

首页首屏保留图片卡片，但新增一个轻输入入口：

```text
说一句今天想怎么玩
例如：我在上海，和刚认识的人，下班后想轻松聊一聊，预算 100 内
```

设计要求：

- 不做长表单。
- 输入框最多 1-2 行。
- 下方保留快捷 chips：
  - 关系：心动 / 朋友 / 家人 / 自己
  - 心情：累了 / 想聊天 / 想拍照 / 想散步
  - 强度：低压力 / 正常 / 多走点
  - 时间：30 分钟 / 90 分钟 / 半天
  - 预算：0 元 / 100 内 / 300 内

用户可以只点 chips，也可以输入一句话。

## 2. 生成前先显示“理解结果”

不要用户点一下就黑箱生成。先给一个轻确认：

```text
我理解的是：
城市：上海
关系：刚认识的心动对象
状态：下班后有点累，想轻松聊天
预算：100 元以内
路线倾向：公开场所、低压力、可随时撤退
```

按钮：

- `就按这个生成`
- `改一下`

这样用户会觉得 App 在理解自己，而不是抽卡。

## 3. 结果页必须解释“为什么适合我”

结果页首屏新增一块短理由：

```text
为什么是这条：
你说今天有点累，所以这条只安排 2 站；
第一站适合自然开口，第二站适合边走边聊；
全程都在公开区域，累了可以直接结束。
```

注意：

- 文案要短。
- 不能变成论文。
- 每条理由都要对应用户输入。

## 4. 路线卡到结果页保持一致

如果用户从卡片开始：

- 结果页主标题沿用卡片标题。
- AI 只能补充副标题、理由和任务。
- 不允许把卡片路线换成完全不同的剧本。

如果用户从一句话开始：

- AI 可以生成新标题。
- 但站点仍来自同城候选池。

## 5. 收藏修成真收藏

V0.9.59 必须修正底部 `收藏`。

最低方案：

- 首页点 `收藏` 后保存路线卡。
- 底部 `收藏` 展示已收藏路线。
- 收藏页可以：
  - 查看路线卡
  - 取消收藏
  - 再次开始

如果做不完：

- 底部名称改为 `灵感`。
- 不再假装是收藏。

## 6. 外测反馈入口

设置页新增：

```text
外测反馈
```

最低方案：

- 一键复制反馈模板。
- 或打开邮箱。
- 或打开问卷 URL。

反馈模板：

```text
手机型号：
城市：
我输入/选择了：
生成结果是否符合：
最不合理的站点：
文字是否太多：
是否愿意发给朋友：
截图：
```

## AI 路线生成架构

### 后端 Endpoint

建议新增：

```text
POST /ai/route/generate
POST /ai/intent/parse
POST /ai/route/feedback
GET  /ai/provider/status
```

也可以先只做一个：

```text
POST /ai/route/generate
```

### Android 请求结构

```json
{
  "locale": "zh-CN",
  "city": "上海",
  "freeText": "我在上海，和刚认识的人，下班后想轻松聊一聊，预算100内",
  "relationship": "date",
  "mood": ["累了", "想聊天"],
  "timeBudgetMinutes": 90,
  "moneyBudgetCny": 100,
  "mobility": "low",
  "candidateStops": [
    {
      "id": "sh-wukang-road",
      "name": "武康路 Citywalk",
      "city": "上海",
      "tags": ["citywalk", "public", "low-pressure"]
    }
  ]
}
```

### 模型输出结构

必须要求 JSON，不接受自由散文。

```json
{
  "title": "下班 90 分钟低压力聊天线",
  "subtitle": "两站都在公开区域，适合轻松开口",
  "understanding": {
    "city": "上海",
    "relationship": "刚认识的心动对象",
    "mainNeed": "轻松聊天，不想太累",
    "constraints": ["100元以内", "90分钟", "低移动压力"]
  },
  "whyThisRoute": [
    "你说下班后有点累，所以只安排 2 站。",
    "第一站适合自然开口，第二站适合边走边聊。",
    "全程都在公开区域，累了可以直接结束。"
  ],
  "selectedStopIds": ["sh-wukang-road", "sh-bund-view"],
  "stopScripts": [
    {
      "stopId": "sh-wukang-road",
      "task": "先散步 15 分钟，找一个橱窗或街角聊第一印象。",
      "conversationPrompt": "今天你最想从哪件小事里恢复一点能量？",
      "exitOption": "如果气氛一般，到这里就可以结束。"
    }
  ],
  "inviteText": "今晚走这条？2站，90分钟，公开场所，轻松聊聊。",
  "safetyNote": "避免偏僻小路，优先公开区域。"
}
```

### 输出校验

后端收到模型输出后必须校验：

- JSON 可解析。
- `selectedStopIds` 全部来自候选列表。
- 所有 stop city 相同。
- 不超过站点数上限。
- 没有危险/冒犯/过度亲密内容。
- 没有虚假评分、热度、官方认证。
- 失败时返回本地生成器结果。

- 新增后端 AI gateway。
- 接入 Kimi/DeepSeek provider。
- 定义 JSON schema。
- 做模型输出校验。
- 做同城、预算、时间、移动压力硬约束。

- 设计 AI 生成测试矩阵。
- 覆盖：
  - 同一输入多次生成是否有变化
  - 不同输入是否明显不同

  - 模型失败是否兜底
  - 低网速/断网是否可用

### 发布合规负责人

- 更新隐私说明：
  - 用户输入会发送到后端/模型服务。
  - 不发送精确定位。
  - 不上传通讯录/相册。
  - 外测数据用途。
- 禁止商店文案夸大“真实 AI 本地导游”。

## V0.9.59 验收标准

### P0

- App 冷启动不闪退。
- 首页仍不是文字墙。
- 用户可以输入一句话或只点 chips。
- App 能显示“我理解的是”。
- 生成结果至少 3 种输入会产生明显不同路线/理由/任务。
- 模型失败时自动回到本地生成器。
- API Key 不存在于 Android 源码、BuildConfig 或 APK。
- 所有路线同城，不混上海/深圳这类跨城错误。
- 底部 `收藏` 不再误导。
- Loading 文案不再写“全球精选”。

### P1

- 结果页解释为什么适合用户。
- 历史记录能保存 AI 路线来源。
- 设置页有外测反馈入口。
- 模拟器普通屏、折叠屏竖屏、折叠屏宽屏截图通过。
- 日志无崩溃。

### P2

- AI 生成中有更明显的阶段动效。
- 完成第一站有奖励反馈。
- 可生成邀请文案。

## 测试用例

### 输入 A：心动低压力

```text
我在上海，和刚认识的人，下班后想轻松聊一聊，预算100内，不想太累。
```

预期：

- 城市：上海
- 关系：心动/刚认识
- 站点：公开、低压力、2 站
- 任务：聊天轻、不过界

### 输入 B：朋友组局

```text
我在广州，三个朋友晚上想随便吃点再散步，预算人均80，不想排队。
```

预期：

- 城市：广州
- 关系：朋友
- 任务：轻互动、拍照、小吃
- 不出现约会文案

### 输入 C：独处回血

```text
我在杭州，今天有点累，想一个人慢慢走，最好安静一点，花钱少。
```

预期：

- 城市：杭州
- 关系：自己
- 任务：低刺激、可暂停
- 文案不劝社交

### 输入 D：家人

```text
我在深圳，周末和爸妈出去，不想走太多，想舒服一点。
```

预期：

- 城市：深圳
- 关系：家人
- 移动压力：低
- 避免酒吧、暧昧、夜间偏僻场景

## 风险与对策

| 风险 | 对策 |
| --- | --- |
| 模型编造不存在地点 | 只允许从候选 POI 选择 |
| API Key 泄漏 | Key 只放后端环境变量 |
| 成本失控 | 后端限流、缓存、短输出、外测白名单 |
| 网络失败 | 本地生成器兜底 |
| 输出太长 | JSON schema 限制字段长度 |
| 生成越界内容 | 关系安全规则 + 后端文本过滤 |
| 用户误以为是真实热度/评分 | 禁止评分、热度、官方认证 |
| 隐私争议 | 设置页和隐私文档明确用户输入会用于生成路线 |
